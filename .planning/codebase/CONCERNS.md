## Tech Debt

| Area | Issue | Files | Impact | Fix Approach |
|------|-------|-------|--------|--------------|
| `IB_PORT` default mismatch | `Config.ib_port` defaults to `5000` but the error message in `connect()` and the env-var memory both say `5001`. The Client Portal Gateway HTTPS port is 5001; port 5000 is the plain-HTTP legacy port. | `trader/config.py:11`, `trader/adapters/ibkr_rest/adapter.py:50` | Every user who relies on the default silently connects to the wrong port and sees only SSL errors, not an authentication failure | Change default from `"5000"` to `"5001"` to match documented behaviour |
| `_resolve_conid` makes one API call per ticker per command invocation | Every `place_order`, `get_quotes`, `get_news`, and `get_option_chain` call hits `/iserver/secdef/search?symbol=` with no in-memory or on-disk cache | `trader/adapters/ibkr_rest/adapter.py:425-442` | A 10-ticker `watchlist show` fires 10 sequential HTTP requests before the market-data snapshot. Slow and burns IBKR rate-limits | Add a `_conid_cache: dict[str, int]` at the adapter instance level; populate on first resolve per ticker per session |
| `Optimizer.grid_search` silently swallows per-combo exceptions | The `except Exception: continue` block means a misconfigured strategy or NaN-heavy DataFrame causes all combos to fail and raises only a generic `RuntimeError` with no diagnostic | `trader/strategies/optimizer.py:27-28` | Agent calling `strategies optimize` gets an opaque failure with no indication of which param combo failed or why | Log the first exception (or count failures); re-raise with context if all combos fail |
| `best_params` not persisted by `strategies optimize` | `optimize` prints best params as JSON to stdout only. `outputs/strategies/best_params.json` exists from the old codebase but the new `optimize` command never writes to it | `trader/cli/strategies.py:157-173`, `outputs/strategies/best_params.json` | Agents that want to reuse tuned parameters across sessions must re-run optimization every time | When `--save` flag is active (already propagated via ctx), write results to `outputs/strategies/best_params.json` in addition to the time-stamped session file |
| `watchlist show --signals` calls `strategy.signal([])` with empty list | The signals branch in `watchlist show` calls `strategy.signal([])` which is not a valid API on `BaseStrategy` (the method is `signals(ohlcv: pd.DataFrame)`). The result is silently discarded | `trader/cli/watchlist.py:160-161` | `--signals` flag on `watchlist show` returns `sentiment_score` but never a real strategy signal; the feature is broken | Replace the placeholder call with a proper `_fetch_ohlcv` + `strat.signals(df)` pattern, same as `strategies signals` command |
| `bracket` CLI command hardcodes `side="buy"` | `orders bracket` always submits the entry as a BUY regardless of intent | `trader/cli/orders.py:92-95` | No way to open a short bracket position via CLI | Add `--side` option; default `buy` but allow `sell` |
| `IBKRTWSAdapter.list_positions` computes `market_value = position * avgCost` | Uses average cost price, not current market price | `trader/adapters/ibkr_tws/adapter.py:155` | Position market value and derived metrics (e.g. pct of NLV) are stale for long-held positions | Use `ib.portfolio()` item's `marketValue` attribute directly instead of computing from avgCost |
| TWS adapter `clientId` is hardcoded to `101` | Every TWS connection uses `clientId=101` with no configurability | `trader/adapters/ibkr_tws/adapter.py:33` | Running two concurrent TWS adapter instances (e.g. CLI + agent) causes IBKR to reject the second with "already in use" | Read from `Config` or env var `IB_CLIENT_ID`; default 101 is fine but must be overridable |

## Security Risks

| Area | Risk | Files | Mitigation | Recommendation |
|------|------|-------|------------|----------------|
| `IBKR_USERNAME` and `IBKR_PASSWORD` stored in `Config` | Broker credentials are loaded into a plain dataclass field at startup and carried in-process for the lifetime of the CLI | `trader/config.py:13-14` | Loaded from env / `.env` only â€” never hardcoded | These fields exist but are not used anywhere in the current codebase. Remove them from `Config` until ibeam auto-login is explicitly implemented; having live credentials in an in-memory object is unnecessary attack surface |
| `verify=False` disables TLS for all IBKR REST calls | `httpx.AsyncClient(verify=False)` suppresses certificate verification globally for the IBKR REST client | `trader/adapters/ibkr_rest/client.py:9` | Documented as intentional for the self-signed Client Portal cert | This is correct for localhost but should be enforced to localhost only. If `IB_HOST` is ever set to a remote address the TLS suppression silently enables MITM. Add a guard: `assert config.ib_host in ("127.0.0.1", "localhost")` when `verify=False`, or load the Client Portal's bundled CA cert instead |
| Auto-confirm of all IBKR warning messages | `_confirm_replies` unconditionally POSTs `{"confirmed": True}` to every IBKR reply message, including potential regulatory warnings about instrument suitability or MiFID compliance | `trader/adapters/ibkr_rest/adapter.py:408-423` | IBKR also applies its own server-side checks | Log the warning message text before confirming so the audit trail captures what was acknowledged; do not confirm silently |
| `AGENT_MODE=autonomous` is the default | The default mode allows the portfolio-conductor agent to place live orders without human review | `trader/config.py:19`, `.claude/agents/portfolio-conductor.md:14` | Agent checks the env var at startup | Document that `AGENT_MODE=supervised` must be explicitly set in `.env` for paper/new accounts; add a check in the conductor that logs a prominent WARNING when running in autonomous mode on a real account |
| No bracket order price direction validation | `place_order` for bracket type does not validate that `take_profit > entry > stop_loss` for a long, or the inverse for a short | `trader/adapters/ibkr_rest/adapter.py:159-180` | IBKR may reject at submission, but the error is surfaced as a generic `ValueError` | Add pre-submission validation in `place_order` before hitting the API: long â†’ `take_profit > price > stop_loss`; short â†’ `stop_loss > price > take_profit` |

## Performance Bottlenecks

| Operation | Problem | Files | Cause | Fix |
|-----------|---------|-------|-------|-----|
| `get_quotes` for N tickers fires N+1 sequential conid lookups | `asyncio.gather` parallelises the N conid lookups, but each lookup is itself a single HTTP call with no cache. For a 20-ticker watchlist that is 20 sequential secdef/search calls before the snapshot | `trader/adapters/ibkr_rest/adapter.py:72-81` | No conid cache | Instance-level `_conid_cache: dict[str, int]`; warm it on first use per ticker |
| Market data snapshot requires a one-second sleep for missing data | A hard-coded `await asyncio.sleep(1)` is inserted whenever any conid returns empty price data | `trader/adapters/ibkr_rest/adapter.py:93` | IBKR subscription model: first call starts stream, second call returns data | This is necessary but should have a configurable timeout (`IBKR_SNAPSHOT_RETRY_DELAY`, default 1s) rather than a magic constant |
| `Optimizer.grid_search` is single-threaded | Grid search iterates all parameter combinations in a single thread with no parallelism | `trader/strategies/optimizer.py:18` | Simple sequential loop | For large grids (e.g. bnf with its 6-parameter space) this is slow. Use `concurrent.futures.ProcessPoolExecutor` with `min(4, os.cpu_count())` workers |
| `strategies signals` calls `yf.download` once per ticker sequentially | Each ticker in the list triggers an independent `yf.download` call inside a `for ticker in ticker_list` loop | `trader/cli/strategies.py:98` | Simple sequential iteration | Use `asyncio.gather` over a thread-pool executor, or batch into a single `yf.download` call with a list of tickers |

## Fragile Areas

| Component | Files | Why Fragile | Safe Modification | Test Gaps |
|-----------|-------|-------------|-------------------|-----------|
| `_resolve_conid` crashes on empty search results | `search[0]["conid"]` raises `IndexError` if IBKR returns an empty list (unknown ticker, market closed, or de-listed symbol) | `trader/adapters/ibkr_rest/adapter.py:431-432`, `adapter.py:112-113` | Wrap in `if not search: raise ValueError(f"No contract found for {ticker}")` to surface a clear error instead of a cryptic IndexError | No test for empty-search-result path |
| Session keepalive is a hard external dependency on `ibeam` | The entire CLI and agent system assumes a running ibeam process is sending `/tickle` every ~60s. If ibeam dies or is not started, `connect()` will retry 8 times then raise, but the scheduled agent will also silently fail to execute | `scripts/start-gateway.sh:34`, `trader/adapters/ibkr_rest/adapter.py:32-51` | No fallback or health probe beyond the connect loop | No tests for ibeam-absent scenarios; no monitoring alert if ibeam process exits |
| `connect()` retry logic sends `/tickle` before auth check on every attempt | The retry loop always POSTs `/tickle` even when the previous attempt returned `authenticated=True` on a subsequent iteration â€” the tickle is safe but the order of operations (tickle-then-check) means the first attempt always makes 2 API calls | `trader/adapters/ibkr_rest/adapter.py:40-43` | Low risk, but confusing if the gateway is healthy | Move the `authenticated` check before the tickle on retries 2+ |
| `watchlists.json` is inside `outputs/` which is gitignored | Watchlists are user state that should survive across machines and git pulls, but `outputs/` is entirely gitignored | `trader/cli/watchlist.py:13-14`, `.gitignore` | Watchlist data is silently lost on a fresh clone | Store watchlists in `.trader/watchlists.json` (the `.trader/` directory is tracked in gitignore pattern `*.jsonl` only, not the directory itself) or add `outputs/watchlists.json` as a `!` gitignore exception |
| MiFID II: US-listed ETFs cannot be traded from EU-domiciled IBKR accounts | `OrderRequest.contract_type` accepts `"etf"` and `orders buy --contract-type etf` will submit the order, but IBKR will reject US ETF trades for EU residents with error code 10168 or similar compliance rejection â€” silently handled as a generic `ValueError` | `trader/models/order.py:16`, `trader/cli/orders.py:37-38` | No regulatory guardrail in the CLI | Document the restriction clearly in `orders buy --help`; ideally surface IBKR's rejection message verbatim rather than wrapping in a generic error |
| `build_context` raises `FileNotFoundError` if `.trader/profile.json` is absent | The agent context builder has no fallback default profile; agents that call `build_context` will hard-fail until the user manually creates the profile file | `trader/agents/context.py:17-21` | Ship a `.trader/profile.json.example` and document the setup step | No test for missing-profile-file path |
| `AgentLog.read_last` reads the entire JSONL file into memory | `path.read_text().strip().splitlines()` loads every log entry ever written before slicing | `trader/agents/log.py:38-43` | Simple implementation; fine for small logs | After 90+ days of autonomous operation the log file could be tens of MB; last-N reads will become slow. Use `collections.deque` with a file seek approach or rotate logs |
| `cancel_order` always returns `True` regardless of outcome | `IBKRRestAdapter.cancel_order` calls DELETE and returns `True` unconditionally, even if the order is already filled or the ID is invalid | `trader/adapters/ibkr_rest/adapter.py:229-233` | DELETE returns HTTP 200 even for already-cancelled orders in some gateway versions | Parse the response body for a success/error indicator; raise on known error shapes |

## Dependencies at Risk

| Package | Risk | Impact | Migration Plan |
|---------|------|--------|----------------|
| `yfinance` (unpinned) | yfinance has a history of silent breaking changes in minor versions. It is used directly in `strategies.py` for OHLCV data for backtesting, signals, and optimization â€” no abstraction layer | All strategy commands (`run`, `signals`, `backtest`, `optimize`) will silently return empty DataFrames on a yfinance API break | Pin to current known-good version in `pyproject.toml`; wrap `yf.download` in a thin `OHLCVFetcher` abstraction so the data source can be swapped without touching CLI code |
| `ibeam` for session keepalive | The system has a hard runtime dependency on the `ibeam` package (`uv run ibeam`) for session persistence. `ibeam` is a community project with infrequent maintenance, and IBKR has changed the Client Portal auth flow before in ways that broke ibeam | Gateway session expires after 24h without keepalive; all CLI commands and scheduled agents fail | Build a minimal fallback keepalive: a simple async loop that POSTs `/tickle` every 55s directly in the adapter, activated when `IBKR_KEEPALIVE=true` is set |
| `ib_insync` (optional, for `ibkr-tws`) | `ib_insync` is an optional extra. The TWS adapter has 3 `NotImplementedError` stubs (`modify_order`, `scan`, `scan_params`, `list_alerts`, `create_alert`, `delete_alert`) that will surface as runtime errors if any agent or CLI command hits those paths after switching brokers | Scanner, alerts, and order modification are silently broken on `ibkr-tws` | Document clearly in CLI `--broker ibkr-tws` help text which commands are unsupported |

## Missing Guardrails

| Gap | Impact | Recommendation |
|-----|--------|----------------|
| No pre-order buying-power / cash check | `place_order` sends any size order to IBKR without verifying that `cash` (not `buying_power`) covers the cost. The portfolio-conductor agent instruction says to use `cash` only, but this is a documentation convention, not an enforced code constraint | An agent arithmetic error or a large quantity can submit an order that IBKR accepts on margin without the system being aware | Add a `max_order_value` check in `place_order`: fetch `account.balance.cash` and refuse if `qty * price > cash * (1 - target_cash_reserve_pct)` |
| No position deduplication before order submission | `place_order` and all CLI order commands do not check for an existing open position in the same ticker before submitting. An agent that re-runs signals without checking open positions can double-enter | Position sizes grow unchecked across repeated agent runs | `place_order` should optionally accept a `check_existing_position=True` flag; the portfolio-conductor should always pass current positions into context before proposing new entries |
| `outputs/` is gitignored but `watchlists.json` needs cross-session persistence | Watchlists are the only truly stateful user data written to `outputs/` that cannot be regenerated. The current gitignore rule silently deletes them on a fresh clone | Watchlists built up over weeks via `watchlist from-scan` are lost on machine migration or re-clone | Move watchlist storage to `.trader/watchlists.json` (survives gitignore) or add `!outputs/watchlists.json` exception to `.gitignore` |
| No test coverage for TWS adapter business logic | The two TWS adapter tests only check importability and class hierarchy; no test exercises `get_quotes`, `place_order`, `list_positions`, or `list_orders` | Regressions in TWS path go undetected | Add unit tests using `unittest.mock` to patch `ib_insync.IB` methods; mirror the pattern in `test_ibkr_rest_adapter.py` |
| No integration test for `_confirm_replies` warning-loop | The IBKR reply confirmation loop (`_confirm_replies`) is a critical path for order submission but has zero test coverage | A change to the response shape from the gateway silently breaks all order placement | Add a unit test with a mock that returns a `{"id": "x", "message": ["are you sure?"]}` response and assert the confirm POST is called |
